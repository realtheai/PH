정책·지원금 AI Agent
정부 정책·지원금 정보를 단순 검색이 아닌 근거 기반 설명 + 개인 자격 가능성 판단까지 제공하는 AI 에이전
트 웹 서비스.

Overview
이 프로젝트는 창업자나 프리랜서가 정부·지자체 창업 지원 정책을 쉽게 탐색하고, 근거 기반의 설명과 자격
판정까지 제공하는 RAG + Agent 기반 정책 판단 보조 서비스이다. 하이브리드 검색(Dense + Sparse)과
LangGraph 기반 상태 관리형 워크플로우를 통해, 사용자의 질문 의도에 따라 정책 문서 기반 답변, 웹 검색
보강 답변, 자격 확인 대화로 흐름을 분기하도록 설계했다.
모든 답변에는 출처를 명시해 신뢰성을 확보했고, 실서비스를 고려해 응답 속도·비용·정확도 간 균형을 핵심
목표로 삼았다.

Key Takeaways
AI 에이전트의 품질은 모델 성능보다 워크플로우 설계에 더 크게 좌우된다
→ LangGraph 기반 상태 관리와 조건부 라우팅으로 응답 일관성과 신뢰성을 확보했다.
RAG 시스템에서 검색 품질은 임베딩 모델보다 데이터 구조와 메타데이터 설계가 핵심
→ 정책 문서 구조를 명확히 정의하지 않으면 하이브리드 검색도 효과가 제한적이었다.
모든 질문을 하나의 흐름으로 처리하는 에이전트는 비효율적
→ Q&A / Eligibility를 분리하고, 근거 충분성 판단 노드를 두어 비용과 응답 시간을 최적화했다.
실서비스형 AI에서는 ‘근거 제시’가 선택이 아니라 필수
→ 모든 답변에 정책 문서·웹 출처를 명시해 설명 가능성과 신뢰성을 확보했다.
Observability 없는 에이전트는 개선이 불가능하다
→ LangSmith 트레이싱과 평가 지표(Groundedness, Citation Rate 등)를 통해 병목과 품질을 정
량적으로 분석했다.

1. 개발 배경 및 목적
최근 정부 및 지자체의 창업 지원 정책은 예산과 사업 수 모두 빠르게 확대되었고, 정책 정보의 접근성과 노출
빈도 역시 알림·통합 포털을 중심으로 크게 개선되었다. 그러나 이러한 변화는 정책의 존재를 인지하는 문제
를 해결했을 뿐, 정책 문구를 해석하고 개인에게 적용 가능한지를 판단하는 문제까지 해결하지는 못했다.
특히 창업자·프리랜서 정책은
자격 조건이 복합적이고
예외·유보 조건이 다수 존재하며
판단 실패 시 시간·기회·재무적 비용이 크게 발생한다.
그럼에도 불구하고 공공 서비스는 책임과 형평성, 비용의 문제로 인해 개별 사용자에 대한 해석이나 판단에
개입하지 않으며, 그 결과 정책 활용 단계에서의 판단 부담과 리스크는 구조적으로 개인에게 귀속된다.
본 프로젝트는 이러한 ‘정책 알림 이후, 실제 의사 결정 이전 단계’의 판단 공백에 주목하여, 정책을 대신 판단
하거나 대행하는 서비스가 아니라 정책 문구를 판단 가능한 단위로 재구성하고, 불확실성을 명시적으로 드러
내는 보조 도구를 제공하는 것을 목표로 한다.

4. 기술적 도전과 해결 과정
본 프로젝트에서는 정책·지원금 정보를 단순히 “찾아주는 시스템”이 아니라, 근거 기반 설명과 자격 가능성 판
단을 함께 제공하는 AI Agent를 목표로 설계했다. 이 과정에서 성능, 신뢰성, 판단 범위 정의라는 세 가지 축
에서 복합적인 기술적 도전에 직면했으며, 아래는 그 해결 과정을 정리한 것이다.

4.1. 검색(Search) 단계의 병목: LLM 기반 Search Agent의 한계
1.1 문제 상황
초기 MVP에서는 정책 검색 또한 LangGraph 기반 Search Agent 워크플로우로 구성되어 있었으며, 질의
이해 → 검색 → 충분성 판단 → 요약 과정에서 LLM을 반복 호출했다.
이 구조는 다음과 같은 문제를 야기했다.
검색 응답 시간이 수 초~수십 초로 증가
검색 트래픽 증가 시 LLM 비용이 선형적으로 증가
“검색”이라는 결정적 작업에 비결정적 요소(LLM 추론)가 개입
1.2 해결 접근
검색을 Agent의 책임에서 분리하고, LLM을 사용하지 않는 결정적 검색 서비스(SimpleSearchService)로
재설계했다.
LangGraph 기반 search_workflow 및 검색 노드 전체 제거
검색 단계에서는 Dense + Sparse 하이브리드 검색만 수행
Agent는 검색 결과를 “사용”하지 “생성”하지 않도록 역할을 축소
1.3 결과
검색 응답 시간: 30초~1분 → 100~500ms
검색 단계에서 LLM 호출 0회
비용·성능·재현성 측면에서 안정적인 구조 확보

4.2. 벡터 검색 단독의 한계: BM25 + RRF 하이브리드 검색 도입
2.1 문제 상황
Dense 벡터 검색(Qdrant)은 의미 유사성에는 강했지만, 정책 도메인 특성상 중요한 정확한 키워드(제도명,
기관명, 정책명) 매칭에서 한계를 보였다.
2.2 해결 접근
Sparse BM25 검색을 병렬로 수행하고, 두 결과를 RRF(Reciprocal Rank Fusion) 방식으로 결합했다.
Dense: 의미 기반 후보 확장
Sparse: 키워드 정합도 보완
RRF: 점수 스케일 차이에 의존하지 않는 안정적 결합
2.3 대안 검토
가중 평균 방식도 검토했으나, 점수 정규화 및 파라미터 튜닝 부담이 커 RRF를 기본 전략으로 채택했다.

4.3. 고정 임계값의 불안정성: 동적 유사도 임계값 전략
3.1 문제 상황
고정 threshold 기반 검색은 질의 유형에 따라 결과 과소/과다 문제가 반복적으로 발생했다.
3.2 해결 접근
검색 결과 수와 키워드 특성을 기준으로 임계값을 자동 조정하는 전략을 도입했다.
결과 부족 시 threshold 하향 → Recall 확보
결과 과다 시 threshold 상향 → Precision 확보
“창업”, “지원금” 등 일반 키워드는 더 낮은 임계값 적용
이를 search_config.py 로 분리하여 운영 중 튜닝 가능하도록 설계했다.

4.4. Q&A 멀티턴 성능 문제: 정책 컨텍스트 캐싱 구조 도입
4.1 문제 상황
Q&A에서 매 질문마다 Qdrant 벡터 검색을 수행하면, 멀티턴 대화에서 지연이 누적되고 동일 정책에 대한 중
복 검색이 발생했다.
4.2 해결 접근
정책 선택 시점에 문서를 1회 로드하고, 이후 Q&A는 캐시에서만 조회하도록 변경했다.
/chat/init-policy 에서 정책 전체 문서 로드 후 PolicyCache 저장

Q&A 단계에서는 Qdrant 미사용 (캐시 조회 5ms 수준)
대화 이력은 ChatCache로 분리 관리
4.3 결과
Retrieval 단계 체감 속도 약 100배 개선
멀티턴 대화 UX 안정화

4.5. 근거 부족 문제: 조건부 웹 검색 폴백 설계
5.1 문제 상황
정책 공고는 최신 변경 사항이 잦아, 내부 DB만으로는 최신성을 보장하기 어렵다. 반면 모든 질문에 웹 검색
을 수행하면 비용과 지연이 증가한다.
5.2 해결 접근
문서 충분성 판단 노드를 통해 근거가 부족한 경우에만 웹 검색을 수행하는 구조를 채택했다.
기본: 문서 기반 답변
부족 시: Tavily 웹 검색 보강
docs_only / web_only / hybrid 프롬프트 분리로 인용 범위 강제

4.6. 자격검증(Eligibility) 도메인의 불확실성: 판단 범위 정의 문제
6.1 문제 상황
자격검증 단계에서는 “무엇을 검증해야 하는지” 자체가 명확하지 않은 문제가 존재했다.
정책마다 자격 조건 표현 방식이 상이
나이, 업력, 지역, 업종 등 조건이 혼합 서술
모든 조건을 동일한 기준으로 판정하기 어려움
6.2 해결 접근
정책 문서 원문을 LLM에 입력하여 자격 조건을 의미 단위 라벨로 추출하는 방식을 시도했다.
LLM을 사용해 자격 조건을 표준 라벨 집합으로 구조화
해당 라벨을 기준으로 Eligibility 프롬프트를 재설계
“판정 가능 / 추가 확인 필요”를 명확히 구분하도록 출력 구조 제한
이를 통해 자격검증 Agent의 역할을 최종 판정이 아닌 구조화된 판단 보조로 명확히 정의했다.

4.7. 자격검증 로직 평가의 한계: 정량 평가 실패와 대안 선택
7.1 문제 상황
자격검증 로직 자체의 성능을 정량적으로 평가하는 단계에서 어려움이 있었다.
어떤 조건이 더 중요한지에 대한 명확한 기준 부재
점수화 기준 설정 시 과도한 주관 개입 위험
정답 라벨이 존재하지 않는 문제 특성
7.2 결과 및 판단
프로젝트 기간 내에 로직 자체의 정량 평가 기준을 확정하지 못했다.
7.3 대안적 선택
이를 억지로 수치화하기보다는,
실제 사용자 테스트를 통해
“이 체크리스트가 도움이 되었는가”
“판정 불가/추가 확인 안내가 신뢰를 해치지 않는가”
를 중심으로 UX 기반 평가를 진행하는 방향이 더 합리적이라고 판단했다.

4.8. 운영·재현성 문제: LangSmith 기반 Observability 도입
LLM 기반 QA·Eligibility 워크플로우는 결과 재현이 어렵기 때문에 LangSimith 기반 Observability를 도입
했다.
LangSmith 트레이싱을 통해 노드·프롬프트·응답 흐름을 추적
session_id, policy_id, feature 단위 태깅으로 디버깅 가능성 확보
이를 통해 “왜 이런 답이 나왔는지”를 사후 분석 가능한 구조로 만들었다.

그 과정에서 기술 선택보다 더 중요한 것이 역할 분리, 판단 범위 정의, 그리고 워크플로우 설계임을 명확히 체
감했다.
Agent를 많이 쓰는 게 아니라, 어디에 쓰지 않을 지를 결정하는 게 중요하단 걸 깨달았다. 또한 멀티턴 Q&A
에서 매 질문마다 벡터 검색을 수행해 지연이 있었다. 이를 해결하기 위해 정책 선택 시점에 문서를 1회 로드
하고, 이후 대화는 캐시된 컨텍스트만 사용하도록 구조를 변경했다. 단순한 구조 변경만으로도 응답 지연 감
소, UX 안정성 개선 등을 달성할 수 있었다. 멀티턴 Agent의 성능은 Retrieval 빈도가 아니라 컨텍스트 전략
에 달려있다는 것도 체감했다.
뿐만 아니라 LLM 답변시 시간 걸리는 문제는 스트리밍 응답을 구현해 해결해 사용자 경험을 향상시켰다.
추후에는 웹 검색에 대한 데이터를 DB에 업데이트하고 정책 데이터에서 자격 요건을 미리 만들어서 DB에 업
데이트해 사용자 호출에 빠른 응답이 가능하도록 할 예정이다.
