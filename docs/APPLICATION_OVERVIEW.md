# 피싱체커 (PhishingChecker) - 애플리케이션 개요

AI 기반 피싱/스미싱 실시간 탐지 시스템 - 의심 메시지를 바로 확인하는 3단계 분석 보조 서비스

---

## Overview

이 프로젝트는 사용자가 받은 문자/메시지가 피싱인지 단순 검색이 아닌 **규칙 기반 + RAG + LLM** 3단계 분석을 통해 판단하고, 근거 기반의 위험도와 대응 권장사항을 제공하는 AI 기반 피싱 탐지 보조 서비스이다.

실제 피싱 사례 7,137건(뉴스 5,800 + 이미지 OCR 1,337)을 pgvector에 임베딩하여 유사 사례 검색(RAG)을 수행하고, OpenAI GPT-4o-mini로 문맥을 분석해 최종 위험도를 산출한다. 모든 판단에는 매칭된 규칙, 유사 사례, LLM 추론이 출처로 명시되어 신뢰성을 확보했다.

| **CI/CD** | GitHub Actions | 자동 배포, 매일 데이터 파이프라인 |

## Key Takeaways

- **규칙 + RAG + LLM 조합**: 단일 방법만으로는 오탐/미탐이 발생 → 3단계 파이프라인이 상호 보완
- **실제 피싱 사례 기반 RAG**: 7,137건 실제 사례로 신규 피싱 패턴에도 대응
- **pgvector HNSW 인덱스**: 전체 DB 검색 ~100ms, 샘플링 없이 정확도 확보
- **데이터 파이프라인 분리**: 크롤링 → OCR → 분류 → 번역 → 임베딩을 GitHub Actions로 자동화
- **보조 도구로서의 역할 정의**: 최종 판단은 사용자, 시스템은 근거 제시에 집중

## 1. 개발 배경 및 목적

피싱/스미싱 메시지는 계좌이체 요구, 택배 사칭, 인증번호 유도 등 형태가 다양해 사용자가 직접 판단하기 어렵다. 기존 안내는 "의심되면 링크 클릭하지 마세요" 수준에 머물러 있으며, 실제로 "이 메시지가 피싱인가"에 대한 즉각적인 판단 도구가 부족했다.

본 프로젝트는 **메시지 입력 → 즉시 위험도 + 근거 + 대응안**을 제공하는 보조 도구를 목표로 한다. 100% 정확도를 보장하지 않으며, 경찰청(182), 금융감독원(1332) 등 기관 확인을 권장하는 구조로 설계했다.

## 2. 데이터 수집 및 DB

### 2.1 데이터 출처

| 출처 | 건수 | 저장소 | 비고 |
|------|------|--------|------|
| 네이버 뉴스 API | - | phishing_news | daily_naver.py |
| 구글 뉴스 | - | phishing_news | daily_google.py |
| Reddit (r/Scams 등) | - | phishing_news | daily_reddit.py |
| 해외 뉴스 (BBC, Reuters) | - | phishing_news | daily_international.py |
| Google Images + OCR | - | phishing_images | daily_google_images.py |

### 2.2 데이터 파이프라인 (8단계)

```
1. 병렬 크롤링 (5개 동시)
   ├─ 네이버 API, 구글 뉴스, Reddit, 해외 뉴스, Google Images
   └─ scripts/daily_*.py

2. 이미지 OCR (Gemini Vision)
   └─ scripts/process_recent_images_ocr.py

3. 콘텐츠 분류 (REAL_CASE / NEWS / NOISE)
   └─ scripts/classify_news_openai.py, classify_images_openai.py

4. 피싱 유형 분류 (11가지)
   └─ scripts/classify_type_openai.py

5. 중복 제거 (의미 기반)
   └─ scripts/deduplicate_data.py

6. 번역 (영어 → 한국어)
   └─ scripts/translate_all_content.py

7. 벡터 임베딩 (OpenAI)
   └─ scripts/embed_with_openai.py

8. Supabase 저장 (phishing_news, phishing_images)
```

### 2.3 DB 스키마 (개요)

- **phishing_news**: source, title, content, url, original_language, classification, phishing_type, embedding
- **phishing_images**: url, ocr_text, translated_text, classification, phishing_type, embedding
- **RPC 함수**: `find_similar_phishing`, `find_similar_phishing_images` (pgvector + HNSW)

1. **규칙 기반**: 300+ 키워드 패턴 매칭, 문맥 기반 점수 조정 (0~100점)

## 4. 기술적 도전과 해결

### 4.1 오탐 문제: 문맥 기반 규칙 조정

- **문제**: "카카오", "인증번호" 같은 단어만으로 피싱으로 판단하면 일상 대화 오탐
- **해결**: Boost Keywords(조합 시 가산), Safe Patterns(일상 표현 시 감점)로 문맥 반영

### 4.2 벡터 검색 품질: 전체 DB vs 샘플링

- **문제**: 샘플링 시 유사 사례 누락
- **해결**: pgvector RPC + HNSW 인덱스로 뉴스 5,800 + 이미지 1,337 전체 검색 (~100ms)

### 4.3 비용 절감: 번역 시점 조정

- **문제**: 모든 데이터 번역 시 비용 증가
- **해결**: 중복 제거 후 남은 데이터만 번역 (약 30% 절감)

### 4.4 크롤링 안정성

- **문제**: newspaper3k의 lxml.html.clean 의존성 분리, 구글 봇 차단
- **대응**: lxml_html_clean 추가, continue-on-error로 파이프라인 진행 보장

## 6. 프로젝트 회고

피싱 탐지는 "완벽한 판정"보다 "근거 있는 보조"에 초점을 맞추는 것이 적절하다. 3단계(규칙/RAG/LLM) 구조는 각 단계의 한계를 상호 보완하며, 사용자에게 "왜 이렇게 판단했는지"를 설명 가능하게 한다.

데이터 파이프라인의 안정성(구글 크롤링 차단, newspaper 의존성 등)은 지속적인 개선 대상이다. 실서비스에서는 정기적인 데이터 품질 점검과 오탐/미탐 사례 피드백 수집이 중요하다.
